  2. Project-k 的定義
    1. 不同環境的 jeforth 使用同一支可抽換的 project-k kernel。
    2. 考慮 proejct-k 可能有 python 版、 rust 版等。
    3. 中心 kernel 定義。

  3. 討論現有的建議。

[ ] Words(a[]) <--- why a[] ?
    A: 這是很老的 code 了。把 word name, xt 等等所有的 properties 用一個 array 送進去。
       這個機關現在看來有點多餘， JavaScript object 的 properties 可以隨時加掛，不需要用
       這樣的技巧。位置在 project-k 的 jeforth.js 裡面，(誰貢獻時間)有空把它改掉。
[ ] panic() <---- why not simply type() ?
    A: jeforth engine or kernel is supposed to be used in embeded systems too.
       In that case, panic() is still necessary while type() has no meaning.
[ ] nexttoken("RegEx") 收 string. 怎麼不直接收 RegEx ?

    A: 既然已經寫好收 string 了，可以不去動它。收 RegEx 感覺怕怕的，string 比較自由。
[ ] value, constant, variable 等都把其值放在 vm.g 之下與
    javascript 共享──沒有任何管理。
    A: 爽哥 解答：把不同 type 再細分擺放。 <== Good idea! 準備有空照這樣改。
[x] 直接執行螢幕上 mark 起來的區塊文字。
A: 不屬 project-k 的工作。 jeforth.3we 的 F4 有類似功能。
   直接執行不太好，讓 user 有機會 edit 補按 才執行。
